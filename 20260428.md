# Guia de Segurança em Aplicações Web

## PARTE 1: Teoria e Conceitos Fundamentais

### 1. A Base da Comunicação: HTTPS, SSL e TLS
* **Detalhes:** O **SSL** e o seu substituto moderno, o **TLS**, são protocolos que criptografam a comunicação entre o navegador do usuário e o servidor web. Quando usamos esses protocolos, a conexão se torna **HTTPS**.
* **O Exemplo:** Imagine que você está acessando seu e-mail no Wi-Fi de um aeroporto. Se o site usar apenas HTTP, qualquer pessoa na mesma rede pode ver a sua senha em texto claro. Se o site usar HTTPS, a sua senha viaja embaralhada e ilegível (criptografada) até o servidor, garantindo a privacidade.

### 2. Projeto Aberto de Segurança em Aplicações Web (OWASP)
A OWASP é uma organização mundial que mapeia as maiores vulnerabilidades da web e ensina como corrigi-las.

**A. Top 10 Web Application Security Risks (Ataque XSS)**
* **Detalhes:** O risco mais famoso é o **XSS (Cross-Site Scripting)**, que é um tipo de injeção de código. Isso acontece quando o site pega um dado enviado pelo usuário (como um nome ou comentário) e o exibe na tela sem realizar nenhuma filtragem.
* **O Exemplo:** Um site pergunta o nome do usuário. O invasor, no lugar do nome, digita um script malicioso: `<script>alert('Hacker')</script>`. Se o site não filtrar isso, o navegador vai achar que esse texto é uma ordem e vai executar o comando.

**B. Top 10 Proactive Controls (Codificação / Encoding)**
* **Detalhes:** Para evitar falhas como o XSS, a OWASP recomenda a **Codificação (Encoding)**. Isso significa transformar os caracteres que formam um código (como os sinais de maior e menor) em um texto inofensivo antes de exibi-los na tela.
* **O Exemplo:** Quando usamos a codificação, o sinal `<` é transformado internamente na entidade `&lt;`. Quando o navegador recebe isso, ele apenas desenha o símbolo na tela, mas perde a capacidade de executá-lo como um comando HTML.

### 3. Mozilla Web Security Guidelines
A Mozilla criou um guia de como configurar os **Cabeçalhos de Segurança (Headers)**. São ordens invisíveis que o servidor manda para o navegador do usuário, ativando escudos de defesa.

* **HSTS (Strict-Transport-Security):** * **Detalhes:** Esse cabeçalho impede conexões inseguras. Ele força o navegador a utilizar **exclusivamente conexões HTTPS criptografadas**, mesmo que o usuário digite "http://" no endereço.
  * **O Exemplo:** `Strict-Transport-Security: max-age=31536000` (Esta ordem diz ao navegador para forçar o HTTPS durante um ano inteiro).
* **X-Frame-Options:** * **Detalhes:** Esse cabeçalho serve para bloquear um ataque chamado **Clickjacking** (Sequestro de Clique). O invasor tenta colocar o seu site verdadeiro dentro de um quadro transparente (iframe) em um site falso, para que o usuário clique em botões perigosos sem perceber.
  * **O Exemplo:** O uso da regra `X-Frame-Options: DENY` impede totalmente que qualquer outro site coloque a sua página dentro de um frame.
* **X-Content-Type-Options:**
  * **Detalhes:** Impede que o navegador tente adivinhar o tipo do arquivo (MIME Sniffing), forçando-o a respeitar o que o servidor declarou.
  * **O Exemplo:** Com a regra `X-Content-Type-Options: nosniff`, se um invasor enviar um arquivo malicioso disfarçado de imagem, o navegador se recusa a executá-lo como um script.

### 4. Content Security Policy (CSP)
* **Detalhes:** O CSP é uma "Lista Branca". Ele é um cabeçalho que diz ao navegador de quais endereços é permitido baixar ou executar scripts e imagens.
* **O Exemplo:** Ao usar a configuração `Content-Security-Policy: default-src 'self'`, você ordena que o navegador só execute scripts que venham do seu próprio domínio. Se um invasor tentar injetar um script vindo de um domínio externo (ex: `site-hacker.com`), o navegador bloqueia a execução na hora.



### 5. Cross-Origin Resource Sharing (CORS)
* **Detalhes:** O navegador tem uma regra nativa que proíbe um site de ler os dados de outro site. O CORS é o cabeçalho usado para abrir exceções a essa regra.
* **O Exemplo:** O CORS controla as permissões de acesso à sua API. O cabeçalho `Access-Control-Allow-Origin: *` possui um asterisco, o que permite que **qualquer site na internet** leia os dados da sua API. Isso é considerado uma falha grave de segurança se a API contiver informações privadas do usuário.

### Experimento 1: O Site com Problema
Crie o arquivo `site_com_problema.py`. Este código ignora as recomendações da OWASP e da Mozilla.

```python
from flask import Flask, request

app = Flask(__name__)

@app.route('/')
def index():
    # PROBLEMA: Exibe o nome da URL sem filtrar (Vulnerável a XSS)
    nome = request.args.get('nome', 'Visitante')
    return f"<h1>Bem-vindo, {nome}</h1><p>Saldo: R$ 5.000,00</p>"

@app.route('/api/dados')
def api():
    # PROBLEMA: Para simular o erro de segurança de dados expostos,
    # vamos liberar o CORS para qualquer um ler (o que é um erro grave)
    from flask import make_response
    res = make_response({"cartao": "4444-5555-6666-7777", "cvv": "123"})
    res.headers['Access-Control-Allow-Origin'] = '*' # ERRO: Qualquer site pode ler
    return res

if __name__ == '__main__':
    app.run(host='127.0.0.1', port=5000)
```
**O que fazer e Resultado Esperado:**
1. **Teste de XSS:** Inicie o servidor e acesse no navegador: `http://127.0.0.1:5000/?nome=<script>alert('Fui Hackeado')</script>`. 
   * *Resultado:* O navegador exibirá um alerta pop-up. Isso prova que o sistema aceitou o código malicioso sem filtrar.
2. **Teste de CORS:** Abra uma aba nova em `https://google.com`. Pressione **F12** para abrir o **Console**. Digite: `fetch('http://127.0.0.1:5000/api/dados').then(r => r.json()).then(console.log)` e dê ENTER.
   * *Resultado:* Os dados do cartão aparecerão no console do Google. O uso do `*` no CORS abriu uma brecha grave.

### Experimento 2: O Site Sem Problema
Crie o arquivo `site_sem_problema.py`. Este código aplica as defesas da OWASP (Encoding) e da Mozilla (Security Headers).

```python
from flask import Flask, request, make_response
import html # Biblioteca para aplicar o Encoding da OWASP

app = Flask(__name__)

@app.route('/')
def index():
    nome = request.args.get('nome', 'Visitante')
    
    # SOLUÇÃO OWASP: Transforma os caracteres de comando em texto
    nome_seguro = html.escape(nome)
    
    html_pagina = f"<h1>Bem-vindo, {nome_seguro}</h1><p>Site Totalmente Protegido.</p>"
    resposta = make_response(html_pagina)
    
    # SOLUÇÃO MOZILLA: Cabeçalhos que ativam o escudo do navegador
    resposta.headers['Content-Security-Policy'] = "default-src 'self'"
    resposta.headers['X-Frame-Options'] = 'DENY'
    resposta.headers['X-Content-Type-Options'] = 'nosniff'
    resposta.headers['Strict-Transport-Security'] = 'max-age=31536000'
    
    return resposta

@app.route('/api/dados')
def api():
    # SOLUÇÃO CORS: Se não enviarmos o header com o asterisco,
    # o navegador bloqueia a leitura externa por segurança padrão.
    resposta = make_response({"status": "Protegido"})
    return resposta

if __name__ == '__main__':
    app.run(host='127.0.0.1', port=5001)
```
**O que fazer e Resultado Esperado:**
1. **Teste de XSS:** Inicie o servidor e acesse: `http://127.0.0.1:5001/?nome=<script>alert('Fui Hackeado')</script>`.
   * *Resultado:* O alerta pop-up NÃO aparece. O texto do script aparecerá escrito na tela inofensivamente. O Console (F12) mostrará um erro vermelho do **CSP** bloqueando o código, demonstrando duas camadas de defesa funcionando.
2. **Teste de CORS:** Volte para a aba do Google, abra o **Console (F12)** e rode o código do `fetch` mudando para a porta `5001`.
   * *Resultado:* Um erro informando que o acesso foi bloqueado pela política de CORS (*Blocked by CORS policy*). Os dados estão protegidos.
