

# SQL Injection – Teoria, Ataque e Prevenção

Curso: Técnico em Informática para Internet
Duração: ~2 horas

---

# 1. Introdução

Uma das vulnerabilidades mais conhecidas em aplicações web é **SQL Injection**.

Ela ocorre quando **dados fornecidos pelo usuário são inseridos diretamente em uma consulta SQL**, permitindo que o usuário altere a lógica da consulta.

Isso pode permitir:

* login sem senha
* acesso a dados confidenciais
* modificação de informações no banco
* exclusão de dados

---

# 2. Como SQL Injection acontece

Considere uma aplicação com login.

Consulta SQL:

```sql
SELECT * FROM users
WHERE username = 'usuario'
AND password = 'senha';
```

Se o programador constrói a consulta concatenando strings, algo assim:

```php
$sql = "SELECT * FROM users
WHERE username = '$user'
AND password = '$password'";
```

o usuário pode enviar **código SQL em vez de apenas dados**.

---

# 3. Exemplo de ataque

Entrada enviada pelo usuário:

```
' OR 1=1 --
```

Consulta resultante:

```sql
SELECT * FROM users
WHERE username = '' OR 1=1 --'
AND password = '';
```

Explicação:

* `1=1` sempre é verdadeiro
* `--` comenta o restante da consulta
* a verificação de senha é ignorada

Resultado: **login realizado sem senha válida**.

# 5. Como prevenir SQL Injection

A forma correta de prevenir é **separar dados da consulta SQL**.

Isso é feito usando **prepared statements (consultas parametrizadas)**.

---

# 6. Prevenção em PHP

Código vulnerável:

```php
$user = $_POST['user'];
$password = $_POST['password'];

$sql = "SELECT * FROM users
WHERE user='$user'
AND password='$password'";
```

Correção usando prepared statements:

```php
$stmt = $conn->prepare(
"SELECT * FROM users
WHERE user = ?
AND password = ?"
);

$stmt->bind_param("ss", $user, $password);
$stmt->execute();
```

Nesse caso:

* os parâmetros são enviados separadamente
* o banco não interpreta o conteúdo como SQL

---

# 7. Prevenção em Node.js

Código vulnerável:

```javascript
const query =
"SELECT * FROM users WHERE email='" +
email +
"' AND password='" +
password +
"'";

connection.query(query);
```

Correção com parâmetros:

```javascript
connection.query(
"SELECT * FROM users WHERE email=? AND password=?",
[email, password]
);
```

A biblioteca envia os valores como **dados**, não como parte do SQL.

---

# 8. Boas práticas adicionais

Para evitar SQL Injection:

* usar **prepared statements**
* nunca concatenar SQL com entrada do usuário
* validar dados recebidos
* limitar permissões do banco de dados
* usar frameworks ou ORM quando possível

---

# 9. Conclusão

SQL Injection é uma vulnerabilidade grave e ainda comum.

Ela ocorre quando aplicações misturam **dados do usuário com código SQL**.

A solução principal é usar **consultas parametrizadas**, garantindo que o banco interprete a entrada apenas como dados.
